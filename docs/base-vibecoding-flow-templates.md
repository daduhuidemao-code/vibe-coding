# Vibe Coding 工作流与基础实践

## 一、文档概述

本文档整合了 Vibe Coding 的核心工作流模板、项目规范、迭代反馈机制和最佳实践，旨在为 AI 辅助编程提供一套完整的方法论和工具集。

### 使用建议

1. **最佳实践**：作为与 AI 协作的指导原则，每次对话前回顾
2. **SPEC 模板**：在项目开始时创建，作为 AI 的执行基准
3. **开发流程**：按照五阶段流程进行项目开发
4. **反馈模板**：在遇到问题或需要优化时使用

---

## 二、最佳实践总结

### 2.1 核心原则

1. **意图驱动**：告诉 AI "做什么"，让它决定"怎么做"
2. **异步协作**：AI 可以在你忙其他事情时工作
3. **信任但验证**：默认信任 AI，但及时验证结果
4. **渐进式调整**：不要期望一次完美，逐步优化
5. **清晰边界**：明确告诉 AI 什么能做、什么不能做

### 2.2 通用检查清单

在发送 Prompt 之前，用这个清单检查一下：

- [ ] 描述了项目背景和目标
- [ ] 列出了具体的功能需求
- [ ] 指定了技术栈和约束条件
- [ ] 明确了设计风格和响应式要求
- [ ] 告诉了 AI 你的技术水平
- [ ] 定义了三层次边界规则
- [ ] 包含了测试和验证计划
- [ ] 制定了提交和迭代策略

### 2.3 常见陷阱

- ❌ 模糊描述："帮我优化一下"
- ❌ 无边界："写一个完整的系统"
- ❌ 强制要求："必须给出正确答案"
- ❌ 忽略测试：只关注功能实现
- ❌ 不设边界：允许 AI 随意修改任何文件

## 六、边界规则（三层次）

### Always（无需询问，直接执行）
- 提交前运行测试
- 遵循命名规范
- 包含错误处理
- 添加必要的注释
- 使用类型安全

### Ask First（需要确认）
- 数据库 Schema 变更
- 添加新依赖
- 修改 API 契约
- 重构核心模块
- 修改 CI/CD 配置

### Never（绝对禁止）
- 提交 secrets（.env、密钥文件）
- 修改 node_modules
- 删除生产数据
- 硬编码敏感信息
- 绕过安全检查

## 四、五阶段开发流程模板

基于 Vibe Coding 的标准工作流：**Explore → Plan → Execute → Verify → Submit**

### 4.1 阶段一：探索项目（Explore）

**目的**：先了解项目当前状态，再制定计划

```
请帮我探索这个项目的结构，并告诉我：
1. 使用了什么技术栈（框架、库、数据库）
2. 文件是如何组织的
3. 有哪些现有的功能模块
4. 配置文件的作用是什么
5. 是否已有可复用的代码或组件

请以结构化的方式输出，便于我理解项目全貌。
```

**应用场景**：
- 刚接手一个新项目
- 需要了解现有代码结构再添加功能
- 不确定项目使用的技术栈

---

### 4.2 阶段二：规划实现步骤（Plan）

**目的**：动手前先规划，减少返工

```
我想添加/修改/实现 [功能名称]。

请帮我规划实现步骤，包括：
1. 需要创建哪些新文件
2. 需要修改哪些现有文件
3. 数据库 Schema 是否需要变更
4. 各个步骤的执行顺序
5. 可能的风险点和注意事项

请以"高级架构师"的视角进行规划，确保方案合理。
```

**详细版（适合复杂功能）**：

```
【目标】
实现 [功能名称]，[一句话描述功能]

【背景】
- 项目状态：[描述项目当前状态]
- 已有资源：[已有相关代码/组件]
- 约束条件：[时间、技术、资源限制]

【功能需求】
1. [功能点1]
2. [功能点2]
3. [功能点3]

【技术方案】
- 前端：[技术选型]
- 后端：[技术选型]
- 数据库：[Schema 变更]

【实施步骤】
| 步骤 | 内容 | 涉及文件 | 依赖 |
|------|------|---------|------|
| 1 | [步骤1] | [文件列表] | [依赖步骤] |
| 2 | [步骤2] | [文件列表] | [依赖步骤] |
| ... | ... | ... | ... |

【风险评估】
- [风险1]：[影响范围和概率]
- [风险2]：[影响范围和概率]

【验证标准】
- [验证点1]
- [验证点2]
```

---

### 4.3 阶段三：执行实现（Execute）

**目的**：按照计划实现功能

```
请按照以下计划实现 [功能名称]：

[粘贴阶段二的计划内容]

要求：
1. 严格按照步骤执行
2. 遵循项目现有的代码风格
3. 不要引入不必要的新依赖
4. 包含完整的错误处理
5. 在关键位置添加注释

完成后告诉我做了哪些改动。
```

**AI 自动拆分能力**：

对于简单任务，你可以直接说：
```
实现 [功能名称]，包含：
1. [功能点1]
2. [功能点2]
3. [功能点3]

技术栈：
- [技术1]
- [技术2]
```

AI 会自动拆分任务并执行。

---

### 4.4 阶段四：测试验证（Verify）

**目的**：确保功能正常工作

```
请帮我测试 [功能名称]，验证以下内容：
1. 核心功能是否正常工作
2. 边界情况处理是否正确
3. 错误处理是否完善
4. 性能是否符合预期
5. 是否有回归问题

测试完成后给出详细的测试报告。
```

**单元测试模板**：

```
请帮我为 [功能/文件] 编写单元测试，使用 [测试框架]。

测试覆盖：
1. [测试场景1]
2. [测试场景2]
3. [测试场景3]
4. [边界情况]
5. [异常情况]

要求：
- 遵循测试金字塔原则
- 使用 Mock 模拟外部依赖
- 确保测试覆盖率达标
- 测试代码风格与项目一致
```

**集成测试模板**：

```
请帮我编写集成测试，验证 [功能模块] 的端到端流程。

测试流程：
1. [步骤1]
2. [步骤2]
3. [步骤3]

验证点：
- 数据从前端到后端的完整流程
- 数据库数据的正确性
- API 响应格式的正确性
- 错误情况下的处理逻辑
```

---

### 4.5 阶段五：提交代码（Submit）

**目的**：保存工作成果，便于后续回溯

```
请帮我提交代码，包含以下改动：
[简要描述本次改动内容]

要求：
1. 使用规范的 commit message（feat/fix/refactor/docs）
2. 添加必要的注释和文档
3. 确保测试通过
4. 更新相关的 README 或文档

提交完成后显示 git log。
```

**自动提交配置**（在 CLAUDE.md 中添加）：

```
"每当完成一个独立功能的开发，或者修复一个 bug 并验证通过后，请自动运行 git commit 提交代码，并生成简洁的中文 commit message。"
```

## 五、持续迭代反馈模板

### 5.1 问题修复模板

```
【问题描述】
[详细描述问题]

【复现步骤】
1. [步骤1]
2. [步骤2]
3. [步骤3]

【期望行为】
[描述期望的行为]

【实际行为】
[描述实际发生的行为]

【错误信息】
[粘贴完整的错误日志]

【相关代码】
[粘贴相关的代码片段]

请分析问题原因并提供修复方案。
```

### 5.2 代码审查模板

```
请以"高级工程师"的身份审查以下代码：

[粘贴代码或指定文件路径]

审查维度：
1. 功能正确性：逻辑是否正确
2. 代码质量：可读性、命名、结构
3. 性能：是否有性能隐患
4. 安全性：是否有安全漏洞
5. 可维护性：是否易于理解和修改
6. 测试覆盖：是否有足够的测试

请给出详细的审查报告和改进建议。
```

### 5.3 性能优化模板

```
请帮我优化 [功能/页面] 的性能，当前问题：
[描述性能问题]

优化方向：
1. 前端优化：[具体方向]
2. 后端优化：[具体方向]
3. 数据库优化：[具体方向]

要求：
- 使用性能分析工具定位瓶颈
- 提供优化前后的对比数据
- 确保不影响现有功能
- 遵循最佳实践
```

### 5.4 安全审查模板

```
请以"安全工程师"的身份审查 [功能/模块]，检查以下方面：

1. SQL 注入风险
2. XSS 漏洞
3. CSRF 防护
4. 认证与授权
5. 敏感数据处理
6. 环境变量管理
7. CORS 配置
8. Rate Limiting

请给出详细的安全审查报告和修复建议。
```
